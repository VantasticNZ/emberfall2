# EMBERFALL — WORKING METHOD (learn, optimize, think ahead between sessions)

> A standing discipline so we improve our PROCESS continuously, not just the game. The time while
> Claude Code runs is not dead time - it's planning/learning/optimizing time.

## THE LOOP (per cycle)
1. BUILD: Claude Code runs a scoped, system-shaped slice (preservation rule always on).
2. WHILE IT RUNS (this is the key habit): Claude(chat) + Van think ahead -
   - Plan the NEXT 1-2 slices so there's never a stall.
   - Capture any new RULES/PRINCIPLES learned from the last playtest into the docs (so we never
     re-learn the same lesson - the anti-circling discipline).
   - Look further ahead at the STORY/EXPLORATION/PLAYTHROUGH depth (the game's soul, not just its
     plumbing).
3. PLAYTEST: Van plays, judges FEEL, reports issues.
4. LEARN: every recurring issue becomes a PRINCIPLE/RULE (batched by system) so one fix kills a
   whole class. Symptoms -> root systems.
5. OPTIMIZE THE PROCESS: prefer fewer, system-shaped sessions over many symptom fixes. Group by
   root cause. Bigger coherent slices when safe.

## OPTIMIZATION PRINCIPLES (how we go faster + better)
- BATCH BY ROOT CAUSE, not symptom (one depth/collision session kills 5 visible bugs).
- SYSTEMS ONCE, CONTENT MANY (build a system robustly; pour content as data).
- PRESERVE ALWAYS (additive/corrective; never regress; push to backup every session).
- VAN'S EYES ARE GROUND TRUTH (reports verify function; only play verifies fun/feel).
- WRITE THE LESSON DOWN (every playtest lesson -> a doc rule -> never repeat it).
- KEEP A LIVING DOC SET in the repo (so Claude Code stops re-deriving from scratch).

## STORY / EXPLORATION / PLAYTHROUGH DEPTH (the game's soul - never neglect)
A technically-solid world is not enough; it must be DEEP + EXCITING. Standing checks we apply to
every area/beat we build (see DEPTH-AND-EXCITEMENT.md for the full framework):
- Does this area have a HOOK (a mystery, a reason to explore, something visible-but-unreachable)?
- Does it reward CURIOSITY (secrets, optional discoveries, "what's over there?")?
- Does it advance or deepen the STORY/characters (not just filler)?
- Does it make the player FEEL something (the tone golden rule)?
- Does it connect to the WEB (a choice, a consequence, a callback, a thread for later)?
- Is there a satisfying GAMEPLAY LOOP here (explore -> challenge -> reward -> new ability/insight)?
If an area is just "technically fine but flat" - it's not done. Depth + excitement are
requirements, not polish.

## STANDING NEXT-UP QUEUE (kept fresh so there's never a stall)
- A (running/next): systemic fixes - depth/collision/accessibility, interaction, karma surfaced.
- Then: the FIRST REGION as careful sub-slices (the Act-2 proof) - built to the depth framework.
- Continuous: capture playtest lessons into rules; keep docs in-repo; push every session.
